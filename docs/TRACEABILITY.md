implements it and the test that exercises it. Every row was earned by reading the implementation and
the test — not by trusting a name, a docstring, or a comment. Where the build does not do what the PRD
says, the row says so and the "Deliberate deviations" section explains why. Re-verified **2026-07-11**
against the repo at `C:\Users\VR\projects\auto-reap`, with the full suite green (`563 passed`).

**Two behaviors worth knowing about the gates** (both deliberate, both visible in `scoring.py` and in the report):
1. **A gate whose input was never measured PASSES with a "not measured" note.** No baseline → `quality_retention` and `domain_regression` report "not measured" and pass. This means a sweep run without a baseline can promote a winner that was never actually gated on quality — the report says so on the row, but the gate column still reads PASS.
2. **`decode_tps` is the only non-blocking gate**, asserted directly in `test_all_gates_pass` (`{g.name for g in results if not g.blocking} == {"decode_tps"}`).

## Deliberate deviations from the PRD

**D1 — The PRD is written for one firm's workload; the build generalizes it to any user via domain packs.**
Rationale (`docs/ACTION_PLAN.md` §1): the PRD's hardcoded assumptions (firm domains, Claude Agent SDK, vr-dispatch, hermes-brain, a 48 GB Ada box) are all made configuration. Domains become YAML packs (`DomainPack`), the frontier model becomes a pluggable provider (`claude-cli` / `anthropic-api` / `openai-compat` / `mock`), the vr-dispatch smoke test becomes `promote.smoke_command`, the hermes-brain page becomes `promote.decision_dir`, and the hardware limits become `Gates` values. **The CPA workload ships intact as one example pack** (`configs/domain-packs/cpa-firm.yaml` — all 8 PRD domains, weighted), alongside `coding-agent` and `general-assistant`. Nothing in the PRD's workload was dropped; it was moved from code into data.

**D2 — FR-2.3's `retained_expert_map` and per-layer saliency stats are not populated.**
`ArtifactManifest.retained_expert_map` is declared and stays `None`; `saliency_stats` carries *expert counts*, not saliency. Rationale (confirmed against the code, `prune/runner.py:read_expert_stats`): REAP's saved checkpoint does not hand back a kept-expert index map — what it *does* carry is the rewritten expert count in the pruned `config.json`. So the manifest records what actually exists and can be verified locally without loading weights: `num_experts_after`, `num_experts_source_field` (which of 5 known MoE config keys it came from), `num_experts_per_tok`, `model_type`. When no known expert-count field is present, the code logs and stores `None` rather than guessing — `test_unknown_shape_returns_none_rather_than_guessing`. The decision is now **documented on the field itself** (`core/records.py:ArtifactManifest.retained_expert_map` explains why it is null and points back here), so the schema is no longer a silent dead end: the field is the declared slot for a map REAP does not emit, kept because a run that *does* emit one (a future upstream, or a metrics-parsing pass) should have somewhere to put it. **Honest assessment:** this is still a genuine reduction in scope from FR-2.3. Recovering the true retained-expert map would mean parsing REAP's own metrics output (the `--record_pruning_metrics_only` path) or diffing router weights, neither of which is implemented.

**D3 — FR-3.1's optional transformers/vLLM runner for HF-checkpoint sanity checks does not exist.**
Confirmed: `runners.py` implements exactly three runners (`LlamaServerRunner`, `OpenAICompatRunner`, `MockRunner`) and `RuntimeCfg.kind` is a `Literal["llama-server", "openai-compat", "mock"]`. Rationale: the PRD's own principle (FR-3.1's first sentence, and the §11 risk row "Quantization compounds pruning loss → evaluate the final GGUF, never only the HF checkpoint") is that the artifact under test must be the artifact that ships. An HF-checkpoint runner would score a model that never reaches production, and pulling in torch/transformers/vLLM would drag the CUDA stack into a package that otherwise installs anywhere and runs its whole test suite offline on Windows. The sanity check it was meant to provide is covered differently: a failed conversion or a broken checkpoint surfaces as a `llama-server` that never becomes healthy, with the exact command to reproduce by hand. It is explicitly listed as PRD-optional.

**D4 — FR-1.5's PII redaction hook is moot: there is no seed-material ingestion path at all.**
Confirmed by search: no config field, CLI flag, or code path accepts user documents as generation input. `ACTION_PLAN.md` §1 planned seed material as optional ("if provided, user points at any redaction command") — **that optional path was never built**, and the default flow described there ("fully synthetic") is the only flow. Every calibration and eval item is either procedurally templated (`procedural.py`) or produced by a provider explicitly instructed to invent everything (`provider_gen.py:_batch_prompt`). The net effect *satisfies* G4 more strongly than redaction would: there is no channel through which client data could reach a cloud call. If seed material is ever added, the redaction requirement returns with it.

**D5 — FR-1.3's near-dup filter defaults to fuzzy matching, not embeddings.**
The PRD specifies an embedding filter blocking cosine ≥ 0.90. The threshold is exactly right (`DataCfg.near_dup_threshold = 0.90`) and the embedding backend is fully implemented (`dedup.py`, cosine over `provider.embed`), but the **default is `dedup_backend: fuzzy`** (rapidfuzz `token_set_ratio / 100 ≥ 0.90`). Rationale: the embedding path needs an embedding-capable endpoint (an LM Studio `/v1/embeddings` model, or an API key), which a first-run user does not necessarily have; the fuzzy backend has zero extra dependencies and no service requirement, and both are tested (`test_fuzzy_blocks_leakage_against_any_calibration_item`, `test_embedding_blocks_leakage`). Set `data.dedup_backend: embedding` + `data.embedding_provider` for the PRD-exact behavior.

**D6 — FR-2.2's remote "teardown" is a documented user step, not an automated one.**
`build_remote_script` deliberately does **not** power the box down (`test_does_not_power_the_box_down`). Rationale, stated in the function's own docstring and in the generated instructions: on most GPU rentals a shutdown *from inside the guest* keeps billing the still-allocated instance while destroying the shell you would use to fetch the result. The budget cap is enforced instead as a `timeout` kill switch around the whole script (clone + build + model download + prune + package), which prints a loud "DESTROY THE INSTANCE YOURSELF" banner on expiry. Destroying the instance is step 5 of the generated `REMOTE_STEPS_<rtag>.md`.

**D7 — FR-2.1's config surface omits `dtype` / `device_map`, because the upstream tool accepts neither.** *(resolved)*
The PRD lists `dtype` and `device_map` among the pruning config's knobs. REAP, at the pinned commit, hardcodes `torch_dtype="auto"` and `device_map="auto"` in `prune.py` and exposes no CLI flags for them (`docs/RESEARCH_BRIEF.md:26`, `docs/REMOTE_GPU.md:121`). They were therefore **removed from `PruneCfg`** rather than carried: as config they were unfalsifiable — accepted, validated, and hashed into `config_hash`, so changing either minted a new run identity (fresh datasets, fresh artifacts, fresh run dir) without changing a single thing REAP did. A knob that silently does nothing while invalidating cached work is worse than an absent one. The absence is pinned by a test (`tests/core/test_config.py::test_prune_cfg_has_no_dead_dtype_knobs`) and explained at the definition site (`core/config.py:PruneCfg`), so it cannot creep back in. If upstream ever exposes the flags, the knobs return with them.

**D8 — The §5 decode-throughput gate compares against a user-set floor, not against the dense 32B daily driver.**
The PRD's advisory target is "≥ current dense 32B daily driver". The build measures decode tok/s per artifact per context and compares it to `gates.min_decode_tps` (unset by default → the gate reports the measured value and always passes). It never discovers or benchmarks the incumbent dense model. Rationale: the incumbent is a property of the user's box, not of the sweep, and the PRD marks this row Advisory. Setting `gates.min_decode_tps` to your current model's measured decode rate reproduces the intent.

## Non-goals (per PRD §3 / §13 — correctly not built)

- **LoRA / SFT "healing" passes** — no fine-tuning code anywhere; REAP is consumed as published, one-shot.
- **Serving infrastructure changes** — promotion copies a GGUF into the LM Studio models dir and stops there.
- **Multi-user productization** — single-operator CLI, local filesystem workspace, no auth, no server.
